# Analisis Penyebab Aplikasi Terasa Berat

## Temuan Masalah Utama

### 1. **Query Database Tidak Efisien - PENYEBAB UTAMA KEMUNGKINAN**

**File: `app/Services/RaporService.php:108-172` (method `nilaiPAS`)**
- Array `$nilaiRaw` di-query satu kali dengan `groupBy('mapel_id')`
- Namun di dalam loop `foreach($mapels as $mapel)`, query `Kktp` dijalankan **satu per mapel** (baris 138-143)
- Ini menciptakan **N+1 query problem**

**File: `app/Http/Controllers/RaporController.php:189-236` (method `simpanPermanen`)**
- Loop untuk setiap siswa, lalu memanggil 4 method service berbeda
- Jika rombel punya 30 siswa, ini artinya 120+ query database

**File: `app/Helpers/RombelHelper.php:17-33` (untuk role ops)**
- Mengambil SEMUA rombel sekaligus dengan `get()` tanpa pagination
- Load `siswas` beserta `ortus` untuk semua rombel sekaligus

### 2. **Bundle Frontend Besar**

**File: `package.json:26-53`**
Dependencies JS yang tidak langsung:
- `xlsx` (~10MB) - hanya untuk export Excel
- `html2canvas`, `html2pdf.js`, `jspdf` (~5-10MB total) - untuk export PDF
- `element-plus` full bundle
- `codemirror` editor (~1MB)

### 3. **Development Tools dalam Production**

**File: `config/querydetector.php:8,14`**
- Query detector aktif dengan threshold 1 (terlalu sensitif)
- Output ke 3 tempat sekaligus: Debugbar + Alert + Log

## Rencana Perbaikan

### Prioritas 1: Perbaiki Query N+1 di RaporService.php
- Eager load semua Kktp sekaligus sebelum loop
- Gunakan mapping array untuk menghindari query berulang

### Prioritas 2: Optimasi simpanPermanen
- Gunakan batch insert/update
- Pertimbangkan queue untuk proses batch besar

### Prioritas 3: Lazy load library frontend
- Dynamic import untuk xlsx, html2pdf.js
- Code splitting di Vite

### Prioritas 4: Disable development tools
- Nonaktifkan query detector di production
- Gunakan APP_ENV check

## File yang Perlu Dibaca Lebih Lanjut
- `app/Models/Sekolah.php` - untuk relasi mapels
- `app/Models/Tp.php` - untuk memahami struktur TP yang di-load
